## Перехват сетевого трафика (Wireshark)

### Запуск Wireshark и проверка захвата трафика

Запуск Wireshark, выбран интерфейс `eth2`.

**Команды:**

```bash
sudo wireshark
```

Проверка:

```bash
ping -c 4 192.168.47.1
```

## Анализ ARP-сообщений

**Очистка ARP-кэша на Kali**

ARP-кэш содержит соответствия между IP-адресами и MAC-адресами устройств в локальной сети. Для того чтобы гарантированно увидеть широковещательный ARP-запрос, кэш был очищен командой:

```bash
sudo ip neigh flush all
```

После выполнения команды таблица ARP стала пустой – при следующей отправке пакета на любой IP-адрес в подсети Kali будет вынуждена выполнить ARP-запрос.

**Отправка ICMP-запроса**

Для инициирования ARP-обмена и последующего ICMP-трафика был отправлен один ping на адрес шлюза (192.168.47.1):

```bash
ping -c 1 192.168.47.1
```


**Анализ ARP-сообщений**

- **Пакет №1 (ARP-запрос)**:  
  - MAC-адрес отправителя – `00:0c:29:84:ee:9a` (Kali).  
  - MAC-адрес получателя – `ff:ff:ff:ff:ff:ff` (широковещательный).  
  - Содержит вопрос: «Кто имеет IP 192.168.47.1? Сообщите 192.168.47.52».  
  - Opcode = 1 (запрос).

- **Пакет №2 (ARP-ответ)**:  
  - MAC-адрес отправителя – `00:0c:29:ca:61:95` (шлюз).  
  - MAC-адрес получателя – Kali.  
  - Сообщает: «IP 192.168.47.1 принадлежит MAC 00:0c:29:ca:61:95».  
  - Opcode = 2 (ответ).

**Анализ ICMP-сообщений**

- **Пакет №3 (Echo request)**:  
  - Тип = 8, код = 0.  
  - Идентификатор `0x743b`, номер последовательности `1`.  
  - TTL = 64.  
  - Длина полезной нагрузки 56 байт.

- **Пакет №4 (Echo reply)**:  
  - Тип = 0, код = 0.  
  - Идентификатор и номер последовательности совпадают с запросом.  
  - TTL = 64.
